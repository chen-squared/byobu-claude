# byobu-claude Development Status

## Implemented

- [x] `bc.sh` main script with all subcommands (init, spawn, send, status, collect, wait, notify, done, logs, kill, cleanup, master, install, help)
- [x] Slash commands for Claude Code (`/bc-delegate`, `/bc-status`, `/bc-collect`, `/bc-orchestrate`, `/bc-kill`)
- [x] Bidirectional communication: main‚Üíchild via `tmux send-keys`, child‚Üímain via `bc.sh notify`
- [x] File-based result passing via `/tmp/claude-comm/tasks/{id}.result.md`
- [x] JSON registry with file locking (`flock`) and exception-safe unlock (`try/finally`)
- [x] Auto-restart wrapper with exponential backoff for API error resilience
- [x] Child reuse: `send` to a completed child auto-resets status and clears stale results
- [x] Safe argument construction: `printf '%q'` quoting for all tmux command strings
- [x] Numeric tmux session name disambiguation (`=` prefix on all targets)
- [x] No `eval` ‚Äî all JSON parsing uses `python3` with `sys.argv`, no shell interpolation

## Potential Improvements (PRs Welcome)

- [ ] **Replace Python JSON helpers with `jq`** ‚Äî each JSON operation currently spawns a Python process; `jq` would be faster but adds a dependency
- [ ] **Structured notification protocol** ‚Äî notifications currently arrive as raw text via `tmux send-keys`; a file-based notification queue would be more robust
- [ ] **Task dependency graph** ‚Äî allow declaring task dependencies so children execute in the correct order
- [ ] **Result file versioning** ‚Äî when reusing children, keep history of previous results instead of overwriting
- [ ] **Web UI / dashboard** ‚Äî a lightweight status dashboard (could be a simple HTML file generated by `bc.sh status --html`)
- [ ] **Session resume after reboot** ‚Äî currently child registry is lost on `/tmp` cleanup; persist to `~/.claude/` for durability
- [ ] **Cross-session communication** ‚Äî allow children in different byobu sessions to be managed by the same master

## Testing Notes

The project has been through 3 rounds of iterative testing:
1. **Round 1**: Basic lifecycle test (spawn ‚Üí send ‚Üí wait ‚Üí collect). Fixed status consistency, argument validation, error handling.
2. **Round 2**: Multi-command integration test. Fixed tmux numeric session ambiguity, registry error propagation, `logs` accuracy.
3. **Round 3**: Independent master E2E test (master process in separate session orchestrating a child through two task cycles). Fixed child reuse status reset, flock exception safety, argument quoting.
