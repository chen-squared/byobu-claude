# byobu-claude Development Status

## Implemented

- [x] `bc.sh` main script with all subcommands (init, spawn, send, status, collect, wait, notify, done, logs, kill, cleanup, master, install, help)
- [x] Slash commands for Claude Code (`/bc-delegate`, `/bc-status`, `/bc-collect`, `/bc-orchestrate`, `/bc-kill`)
- [x] Bidirectional communication: main→child via `tmux send-keys`, child→main via `bc.sh notify`
- [x] File-based result passing via `$BC_COMM_DIR/tasks/{id}.result.md`
- [x] JSON registry with file locking (`flock`) and exception-safe unlock (`try/finally`)
- [x] Auto-restart wrapper with exponential backoff for API error resilience
- [x] Child reuse: `send` to a completed child auto-resets status and clears stale results
- [x] Safe argument construction: `printf '%q'` quoting for all tmux command strings
- [x] Numeric tmux session name disambiguation (`=` prefix on all targets)
- [x] No `eval` — all JSON parsing uses `python3` with `sys.argv`, no shell interpolation
- [x] Multi-master isolation: each master gets auto-namespaced `BC_COMM_DIR` and unique `master_id`; children notify correct master via session+window; env leak prevented with `env -u`

## Potential Improvements (PRs Welcome)

- [ ] **Replace Python JSON helpers with `jq`** — each JSON operation currently spawns a Python process; `jq` would be faster but adds a dependency
- [ ] **Structured notification protocol** — notifications currently arrive as raw text via `tmux send-keys`; a file-based notification queue would be more robust
- [ ] **Task dependency graph** — allow declaring task dependencies so children execute in the correct order
- [ ] **Result file versioning** — when reusing children, keep history of previous results instead of overwriting
- [ ] **Web UI / dashboard** — a lightweight status dashboard (could be a simple HTML file generated by `bc.sh status --html`)
- [ ] **Session resume after reboot** — currently child registry is lost on `/tmp` cleanup; persist to `~/.claude/` for durability

## Testing Notes

The project has been through 3 rounds of iterative testing:
1. **Round 1**: Basic lifecycle test (spawn → send → wait → collect). Fixed status consistency, argument validation, error handling.
2. **Round 2**: Multi-command integration test. Fixed tmux numeric session ambiguity, registry error propagation, `logs` accuracy.
3. **Round 3**: Independent master E2E test (master process in separate session orchestrating a child through two task cycles). Fixed child reuse status reset, flock exception safety, argument quoting.
4. **Round 4**: Multi-master isolation test (two masters running simultaneously). Fixed: (a) BC_COMM_DIR auto-namespaced per session, (b) master_id collision detection, (c) notify targeting wrong master (now uses dedicated bc-master window per session), (d) env variable leak prevention (`env -u`), (e) quoting of master_id as command-line argument.
5. **Round 5**: Post-Round-4 bug fix. Found and fixed: (a) `cmd_init` wrote 4-field master.lock (missing session/window), causing `notify` to fail — now writes 6 fields matching `cmd_master` format; (b) `cmd_master-init-wrapper` double-consumed master_id arg, causing `cmd_wrapper` to use a claude flag as master_id — fixed by passing all args through; (c) duplicate `local master_window` in `cmd_master`; (d) hardcoded `/tmp/claude-comm/` in child system prompt and slash commands — now uses `$BC_COMM_DIR`.
6. **Round 6**: Real byobu end-to-end test. Launched two master sessions (`bc-master-a`, `bc-master-b`) simultaneously, each spawning children and running full lifecycle (init → spawn → send → wait → collect → cleanup). Confirmed: multi-master isolation works (each master's tasks and results are fully separated), child notify reaches correct master, child reuse works, auto-restart wrapper handles API errors gracefully.
